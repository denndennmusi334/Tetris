# オンライン対応テトリス

## 概要
このテトリスは、同じLAN内（Wi-Fiや有線LANの同じネットワーク内）のプレイヤー同士でオンライン対戦ができるゲームです。
T-SpinやREN（コンボ）など、本家テトリスでおなじみのシステムも一通り実装しています。

## 仕様

### 操作方法（キーボード）
- **左移動**：`A`
- **右移動**：`D`
- **左回転**：`Q`
- **右回転**：`E`
- **ハードドロップ**：`W`
- **ソフトドロップ**：`S`

※現在はオンライン対戦モードのみの実装となっています。

---

## 遊び方（接続方法）

同じLAN内にいる2台のPCでゲームを起動します。

1. **ホスト（対戦を待ち受ける側）**
   - ゲームメニューから「ホストとして起動」を選択します（または設定ファイルでホストに設定）。
   - 自身のプライベートIPアドレス（例: `192.168.x.x`）を対戦相手に伝えてください。
2. **クライアント（接続する側）**
   - ゲームメニューから「クライアントとして起動」を選択します。
   - 接続先入力画面、または設定ファイル（`config.json` 等）に、ホストから聞いたIPアドレスを入力して接続します。
3. **対戦開始**
   - 接続が確立されると、自動的に対戦画面へ遷移します。

## 使用ライブラリ

- **Dxライブラリ**
  - 主に描画処理とFPS管理に使用しています。
  - *※近いうちに、描画以外のロジックをDxライブラリから独立させ、依存度を下げる設計に変更する予定です。*
- **httplib**
  - 対戦に入る前のマッチング機能を行うために使用しているライブラリです。
  - 対戦前は何度もデータを送る必要がないため、単方向通信を使用しています。
- **IXWebSocket**
  - 対戦中のリアルタイム通信（WebSocket通信）を行うために使用しているライブラリです。
- **nlohmann/json**
  - JSONファイルの読み込み・書き出しを行うライブラリです。
  - 本作では、通信時のデータ送受信を基本的にJSONフォーマットで行っているため、そのパースや生成に使用しています。

## ネットワーク設計・通信処理の詳細 (Networking & Architecture)

本作のネットワーク周りは、リアルタイム性と確実性を両立するため、**HTTP通信(ロビー・マッチング管理)** と **WebSocket通信(対戦中のリアルタイム同期)** を組み合わせたハイブリッドなアーキテクチャを採用しています。

### 1. 通信プロトコルの使い分け
シチュエーションに応じて最適なプロトコルを選択し、ネットワーク負荷と実装の複雑性を制御しています。

- **HTTP（cpp-httplib）によるセッション管理**
  部屋の作成 (`/init`)、部屋一覧の取得 (`/roomlist`)、入室 (`/join`)、退室 (`/logout` / `/dismantle`) といった、リアルタイム性を必要としない確実なステート遷移には、要求・レスポンスが明確なHTTPを採用しています。
- **WebSocket（IXWebSocket）によるリアルタイム同期**
  入室完了後 (`Connect()`) は、遅延（レイテンシ）を最小限に抑えるため、永続的な接続を確立するWebSocketに切り替え、JSONフォーマット（`nlohmann/json`）による双方向リアルタイム通信を行います。

### 2. 受信メッセージの処理とゲームループへの統合
WebSocketのメッセージ受信は内部のバックグラウンドスレッドで行われるため、ゲームのメインスレッド（メインループ）と安全にデータをやり取りする仕組みを構築しています。

1. **非同期受信とキューイング**:
   `ws.setOnMessageCallback` 内でデータを受信した際、メインループをブロックしないよう、即座にパースしてスレッドセーフな受信キュー（`recvQueue`）にプッシュします。
2. **メインスレッドでの一括処理**:
   毎フレームの `NetworkManager::Update()` 内で、キューに溜まったメッセージを全て取り出し、ゲームの進行状況に合わせて `HandleMessage()` で安全に処理します。

## 堅牢性の向上と防御的プログラミング

オンラインゲームにおいて、通信データの不正やパケットロスによるクラッシュは致命的です。本作では、サーバーから送られてくるJSONデータに対して徹底したバリデーション（型・境界値チェック）を行っています。

- **配列サイズ・境界値の厳密なチェック**
  `map` データの受信時、サーバー側のデータサイズがクライアント側の想定（フィールド幅・高さ）を超えていた場合にメモリ不正アクセスを引き起こさないよう、`std::min` を用いて最大値をクリッピングしています。
- **厳格な型安全性の確保**
  `jsonMap[y][x].is_number_integer()` など、ネストされたJSONの各要素が想定通りの型であるかを1マスずつ検証し、不正なデータや欠損があった場合は `-1`（空マス）を安全に補完する設計にしています。

## 直面した課題と解決策（スレッドセーフなデータ同期）

### 課題：非同期データ書き込みによるデータ競合（Data Race）
開発初期、ゲームメインループ内の描画処理やロジック処理が「読み書きしている最中のバトルデータ（`battleDataH`）」に対して、WebSocketスレッドから非同期に新しいホスト（対戦相手）のデータが届き、そのまま上書きされてしまう場面がありました。
これにより、タイミングによってデータの不整合やメモリの破損（ブロックの配置データが化ける、クラッシュするなど）が発生するリスクが生じました。

### 原因分析
IXWebSocketの受信コールバックは**メインスレッドとは異なる別スレッド（通信スレッド）**で動作します。
`HandleMessage` 内で、通信スレッドから通知された対戦相手の盤面情報やミノの状態（`battle_data_h`）をグローバルなゲーム状態へ同期する際、メインスレッドがそのデータを参照（読み込み）しているタイミングと衝突していたことが原因でした。

### 対策：ミューテックス（Mutex）による排他制御
この問題を解決するため、バトルデータ専用のミューテックス `battleMutex` を導入しました。

```cpp
// HandleMessage内での対策コード
if (type == "battle_data_h")
{
    BattleDataH data = { 0 };
    
    // ... (受信データのパースとローカル構造体への安全な展開) ...

    // lock_guardを使って、battleDataHへのアクセスを保護
    // 複数のスレッドが同時にアクセスしてデータ競合が発生するのを防ぐ
    std::lock_guard<std::mutex> lock(battleMutex);
    battleDataH = data; 
}
```

## 直面した課題と解決策（通信ラグと同期ズレへのアプローチ）

リアルタイムな操作性が求められるテトリスにおいて、ネットワークのレイテンシ（遅延）と画面の同期ズレは最も大きな課題でした。本作の完成に至るまで、以下の3つのアプローチを経て、通信量と操作感の最適解を導き出しました。

### 【フェーズ1】完全ホスト管理型（入力を送り、結果を待つ）
* **設計**: クライアント側のキー入力をすべてホスト（サーバー）へ送信し、ホスト側で移動・回転等の物理計算を行った結果をクライアントに送り返して画面を描画する方式。
* **課題**: 入力してから画面に反映されるまでに「クライアント→ホスト→クライアント」の往復時間（RTT）分の**大幅なラグ**が生じ、テトリス特有の素早い操作（高速な横移動や回転）が完全に損なわれてしまいました。

### 【フェーズ2】クライアント側での先行予測計算（ローカル処理の導入）
* **設計**: ラグを解消するため、クライアント側でもホストからの応答を待たずに、キー入力に対して即座にローカルでミノの移動計算（クライアントサイド・予測）を行う方式に変更。
* **課題**: 操作のラグは解消されたものの、ネットワークの揺らぎや処理タイミングの僅かな差によって、ホストとクライアント間で**激しい同期ズレ**が発生。ミノの位置がワープしたり、本来回るはずの場所で回らなくなるといった致命的な問題が起きました。

### 【フェーズ3】最終解決策：イベント駆動型のリモート状態同期（最適化）
* **結論**: 「操作中のミノ」と「固定された盤面」で同期の頻度と粒度を分ける、**イベント駆動型の同期アルゴリズム**を考案・実装しました。

1.  **通常操作時（低負荷・低同期）**
    移動や回転などの細かな操作中は、毎フレームの盤面全体の同期をやめ、お邪魔ブロックの送受信や、演出に必要な最小限のデータのみを軽量に通信します。これにより、クライアント側は100%ローカルの快適な操作感（レイテンシゼロ）を維持できます。
2.  **ミノ固定時（厳密な確定同期）**
    プレイヤーがミノを「ハードドロップ」または「ロックダウン（固定）」した瞬間にのみ、確定した盤面情報（`map` データ）をホスト・クライアント間で相互に送信し合います。

#### ■ この設計によるメリット
テトリスというゲームの性質上、「ミノが動いている最中」はお互いの画面に多少のズレ（コンマ数秒の描画の差）があってもゲームプレイに支障はありません。しかし、「盤面が確定する瞬間」だけは絶対にズレてはいけません。
この性質に注目し、**「普段は自由（ラグなし）、要所だけ厳密（ズレなし）」**というメリハリをつけたデータ構造にしたことで、LAN内における非常に滑らかでストレスのない対戦環境を実現しました。
